Streaming CSV Parse

128 KB BufferedReader · OpenCSV · file stays on disk
❌ Problem If you call file.getBytes() or load the whole CSV into a String, an 800 MB file puts 800 MB on the heap — instant OutOfMemoryError.

Spring's multipart resolver streams the upload to a temp file on disk (controlled by file-size-threshold=1MB in application.properties). The raw InputStream from MultipartFile.getInputStream() reads from that temp file — meaning at most a few KB of CSV data are decoded in the JVM heap at any moment.

✅ Solution Wrap the stream in a 128 KB BufferedReader, feed it to OpenCSV's CSVReaderBuilder with withSkipLines(1) to drop the header. Only the current line is ever decoded in memory.

🔍 Deep-dive: spring.servlet.multipart.file-size-threshold

Spring Boot's multipart support is backed by either Apache Commons FileUpload (Servlet 3 mode) or the Servlet 3.0 Part API (default in Spring Boot 3). Both share the same threshold concept: incoming bytes are held in a memory buffer until the threshold is crossed, at which point the remainder is spooled to a temporary file on disk. The property that controls this boundary is:

ℹ️ How the threshold works spring.servlet.multipart.file-size-threshold sets the number of bytes that can sit in JVM heap before Spring switches to disk.

Default value: 0B (zero bytes) — meaning Spring writes to a temp file immediately for every upload, regardless of size.

If you set it to 10MB, files smaller than 10 MB are kept in a byte[] in memory; files larger are spooled to disk once 10 MB have been buffered. For large CSV processing, always keep this at 0B or 1B so even tiny files go straight to disk and you can stream from the very first byte.
Property What it controls Recommended value
file-size-threshold Bytes buffered in heap before spooling to temp file 0B or 1B — never buffer in memory
max-file-size Maximum size of a single uploaded file (hard reject above this) 2048MB (or as large as you accept)
max-request-size Maximum size of the entire multipart request (all parts combined) 2048MB (match or exceed max-file-size)
location Directory for temp files when spooled to disk Fast local SSD path, e.g. /tmp/uploads
enabled Enables Spring's multipart resolver (must be true) true

When the threshold is set to 0B, here is exactly what happens inside Tomcat / Spring when a file upload arrives:

  • 1 The HTTP request body arrives over the socket. Tomcat reads it in 8 KB chunks into a small socket receive buffer — this is OS-level, not JVM heap.
  • 2 Because file-size-threshold=0B, Tomcat's DiskFileItem immediately opens a temp file (in java.io.tmpdir or the configured location) and writes every incoming chunk straight to disk. No byte array is allocated on the JVM heap for the file content.
  • 3 Your controller receives a MultipartFile. When you call getInputStream(), Spring returns a FileInputStream backed by that temp file on disk — not a ByteArrayInputStream from memory.
  • 4 You wrap that stream in a BufferedReader with a 128 KB buffer. Only 128 KB of file data resides on the JVM heap at any instant — regardless of whether the file is 1 MB or 10 GB.
  • 5 When the request ends, Spring calls MultipartFile.transferTo() cleanup or the servlet container deletes the temp file automatically.
application.properties — full multipart configuration
# ── Multipart / file-upload settings ──────────────────────────────────────

# CRITICAL: set to 0B so Spring never buffers ANY part of the file in heap.
# Default is 0B in Spring Boot 3, but make it explicit.
spring.servlet.multipart.file-size-threshold=0B

# Maximum size for a single uploaded file (HTTP 413 if exceeded)
spring.servlet.multipart.max-file-size=2048MB

# Maximum total multipart request size
spring.servlet.multipart.max-request-size=2048MB

# Temp directory for spooled files. Use a fast local SSD path.
# Defaults to java.io.tmpdir if not set.
spring.servlet.multipart.location=/tmp/csv-uploads

# Must be true (it is by default) to enable the multipart resolver
spring.servlet.multipart.enabled=true
⚠️ What happens if you forget to set the threshold If file-size-threshold is set to a high value (e.g. 50MB), every file under 50 MB is buffered entirely in a byte[] on the heap. A typical CSV batch job that receives dozens of concurrent 30 MB uploads would allocate 30 MB × concurrent uploads in heap before any processing starts. With 20 concurrent requests that is 600 MB just in upload buffers — enough to exhaust a 512 MB heap immediately.
UploadController.java — controller that receives the stream
@PostMapping("/upload")
public ResponseEntity<String> uploadCsv(
        @RequestParam("file") MultipartFile file) {

    // ✅ file.getInputStream() returns a FileInputStream backed by a
    //    temp file on disk — NOT a ByteArrayInputStream from heap.
    //    This is guaranteed ONLY when file-size-threshold=0B.
    try (InputStream is = file.getInputStream()) {
        csvProcessingService.process(is);
    } catch (IOException e) {
        throw new RuntimeException("Stream open failed", e);
    }
    return ResponseEntity.ok("Processing started");
}

         new InputStreamReader(inputStream, StandardCharsets.UTF_8),

Projection-Based Upsert Lookup

CustomerIdProjection · only (externalId, id) tuples loaded
❌ Problem To decide whether to INSERT or UPDATE, we need to know if each externalId already exists in the database. The naive approach loads full Customer entities (all 8 fields, Hibernate state, dirty-check snapshots) just to read one field — wasting ~300 B per row.

A Spring Data projection is a plain Java interface whose getter names match JPQL aliases. Spring Data materialises it as a tiny JDK dynamic proxy (~200 bytes) rather than a full entity (~500 bytes including Hibernate's internal snapshot array). For a batch of 500 rows, that's a saving of ~150 KB per batch — and across 20 000 batches (10 M rows) it means gigabytes of avoided allocation.

✅ Solution Define a CustomerIdProjection interface with two getters. Write a JPQL query that selects only those two columns aliased to match. Spring Data does the rest.
CustomerIdProjection.java
/** Two-field projection — ~200 B proxy vs ~500 B full entity */
public interface CustomerIdProjection {
    String getExternalId();
    Long   getId();
}
CustomerRepository.java
// JPQL aliases must match the interface getter names (minus "get", lower-cased)
@Query("SELECT c.externalId AS externalId, c.id AS id " +
       "FROM Customer c WHERE c.externalId IN :externalIds")
List<CustomerIdProjection> findIdsByExternalIdIn(
        Collection<String> externalIds);
BatchUpsertService.java — using the projection
// ONE SELECT per batch — not N selects for N rows
Map<String, Long> existingIdMap = customerRepository
        .findIdsByExternalIdIn(externalIds)
        .stream()
        .collect(Collectors.toMap(
                CustomerIdProjection::getExternalId,
                CustomerIdProjection::getId));

// Stamp DB id → Hibernate issues UPDATE; no id → INSERT
for (Customer c : batch) {
    Long dbId = existingIdMap.get(c.getExternalId());
    if (dbId != null) c.setId(dbId);  // UPDATE
    // else: id = null → INSERT
}

EntityManager flush() + clear()

Release first-level cache · entities become GC-eligible immediately
❌ Problem — Hibernate's silent memory leak Hibernate's first-level cache (the persistence context / session) keeps a reference to every entity it has ever touched in the current session. Without explicit clearing, calling saveAll() 20 000 times means 10 million Customer objects accumulate in memory — guaranteed OOM.

After saveAll(), two calls eliminate the leak:

  • 1 entityManager.flush() — forces Hibernate to send any remaining dirty SQL to JDBC. This is a safety net; usually saveAll() already flushed, but explicit flushing ensures nothing is left buffered.
  • 2 entityManager.clear() — detaches all entities from the session. Hibernate drops every object reference it holds. The 500 Customer objects and 500 projection proxies from the current batch become immediately eligible for garbage collection.
⚠️ Spring Self-Invocation Pitfall @Transactional works via AOP proxies. If you call this.upsertBatch() from within the same Spring bean, the proxy is bypassed — no transaction is created and entityManager.flush() throws "No EntityManager with actual transaction available". The fix: put the batch logic in a separate Spring bean (BatchUpsertService) so every call goes through the proxy.
BatchUpsertService.java
@Service
public class BatchUpsertService {

    @PersistenceContext             // transaction-scoped EntityManager
    private EntityManager entityManager;

    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public long[] upsertBatch(List<Customer> batch, ...) {

        customerRepository.saveAll(batch);  // JDBC batch INSERT/UPDATE

        // ── Technique 3 ─────────────────────────────────────────
        entityManager.flush();   // flush remaining dirty state to JDBC
        entityManager.clear();   // detach ALL entities → eligible for GC
        // ────────────────────────────────────────────────────────

        return new long[] { inserts, updates };
    }
}

Batch Size Capped at 500

Lower per-TX peak heap · tuned Hibernate JDBC batch_size
❌ Problem A large batch size (e.g. 5 000) means 5 000 entity objects in memory simultaneously before saveAll() can release them. At ~300 bytes each that's 1.5 MB of entity objects per batch — plus Hibernate's snapshot copies, JDBC parameter arrays, and collection overhead.

Batch size controls two separate things that must be kept in sync:

  • 1 Application batch — how many rows are accumulated in the List<Customer> batch before calling upsertBatch(). Set via csv.processing.batch-size=500.
  • 2 JDBC batch — how many SQL statements Hibernate bundles into one network round-trip. Set via hibernate.jdbc.batch_size=500. Must equal or be a divisor of the application batch so every call maps to exactly one JDBC batch.
✅ Tuning result 500 rows × ~300 B = 150 KB peak heap per transaction. Each batch completes in one JDBC round-trip. Throughput is unchanged vs 1 000 batch size because SQLite's bottleneck is disk I/O, not round-trips.
application.properties
# Application batch size (rows accumulated before one upsertBatch() call)
csv.processing.batch-size=500

# Hibernate JDBC batch — must match application batch for efficiency
spring.jpa.properties.hibernate.jdbc.batch_size=500
spring.jpa.properties.hibernate.order_inserts=true
spring.jpa.properties.hibernate.order_updates=true
spring.jpa.properties.hibernate.batch_versioned_data=true

# Disable stats — removes per-statement overhead objects
spring.jpa.properties.hibernate.generate_statistics=false

Heap Telemetry Every 100 Batches

MemoryMXBean · no profiler needed · verify flat heap live
❌ Problem Without visibility into heap usage, you cannot tell whether your optimisations are working. Attaching a profiler to a production or staging JVM is invasive and sometimes not possible.

Java's built-in MemoryMXBean (available via ManagementFactory.getMemoryMXBean()) gives real-time heap figures — used, committed, and max — with zero external dependencies. Logging these every 100 batches (50 000 rows) adds one line to the application log and lets you verify the heap is flat without attaching any external tool.

✅ Solution Call logHeap() from the main batch loop every HEAP_LOG_EVERY_N_BATCHES = 100 batches. If heap is rising monotonically, the flush()+clear() is not working — this is your early-warning signal.
CsvProcessingService.java
private final MemoryMXBean memBean = ManagementFactory.getMemoryMXBean();
private static final int HEAP_LOG_EVERY_N_BATCHES = 100;

private void logHeap(ProcessingJob job, String phase) {
    long used      = memBean.getHeapMemoryUsage().getUsed()      / (1024 * 1024);
    long committed = memBean.getHeapMemoryUsage().getCommitted()  / (1024 * 1024);
    long max       = memBean.getHeapMemoryUsage().getMax()        / (1024 * 1024);
    int  pct       = (int)((used * 100) / max);

    log.info("[Job {}] [{}] Heap: used={} MB | committed={} MB | max={} MB  ({}% used)",
             job.getJobId(), phase, used, committed, max, pct);
}

// Called from flushBatch() every 100 batches
if (batchNo % HEAP_LOG_EVERY_N_BATCHES == 0) {
    logHeap(job, "batch-" + batchNo);
}

Bonus: G1GC Tuning for 512 MB

start.bat / startup flags · early GC trigger · GC log

The JVM flags below lock the heap to exactly 512 MB and tune G1GC to collect early and often rather than letting the heap fill up before a major GC pause.

start.bat
java \
  -Xms512m -Xmx512m                        # hard cap — no expansion allowed
  -XX:+UseG1GC                              # G1 GC (default in Java 9+)
  -XX:MaxGCPauseMillis=100                  # target max pause 100 ms
  -XX:InitiatingHeapOccupancyPercent=30     # trigger concurrent GC at 30% full
  -XX:G1HeapRegionSize=4m                   # 4 MB regions suit 512 MB heap
  -XX:+ExplicitGCInvokesConcurrent          # System.gc() stays concurrent
  -Xlog:gc*:file=logs/gc.log:time,uptime,level  # GC log for analysis
  -jar target/csv-upload-demo-1.0-SNAPSHOT.jar
FlagWhy it matters at 512 MB
-Xms512m -Xmx512mFixed heap = no slow heap expansion pauses mid-processing
-XX:InitiatingHeapOccupancyPercent=30Default is 45% — lowering to 30% starts concurrent marking earlier, preventing promotion failure with flat but active heap
-XX:G1HeapRegionSize=4m512 MB / 4 MB = 128 regions — G1's sweet spot for this heap size
-Xlog:gc*:file=logs/gc.logGC log lets you verify pause times and collection frequency post-run
