📬 Queues & Deques
ArrayDequejava.util Non-TS Resizable circular-array deque. Preferred stack and queue implementation. Circular array with head/tail pointers; doubles capacity when full. No capacity limit (unlike ArrayBlockingQueue). Null elements not allowed. offer/poll head/tail O(1)† get O(1)
  • Stack replacement (faster than Stack/LinkedList)
  • FIFO queue replacement (faster than LinkedList)
  • BFS/DFS algorithms
  • Concurrent access
  • Bounded-capacity blocking scenarios
PriorityQueuejava.util Non-TS Min-heap that returns elements in priority order. Binary min-heap over an array. peek/poll returns the smallest element. Elements must be Comparable or a Comparator supplied. Does not allow nulls. offer O(log n) poll O(log n) peek O(1)
  • Dijkstra / A* / task scheduling
  • Top-K problems
  • Event-driven simulations
  • Concurrent access without synchronization
  • When FIFO ordering is required (use ArrayDeque)
ConcurrentLinkedQueuejava.util.concurrent Thread-Safe Unbounded, lock-free FIFO queue. Michael-Scott non-blocking linked queue algorithm using CAS. size() is O(n) — avoid it. No blocking operations. offer/poll O(1) size O(n) lock-free
  • High-throughput producer/consumer (non-blocking)
  • Work queues where blocking is undesirable
  • When backpressure/bounded capacity is needed
  • When blocking await is required
ConcurrentLinkedDequejava.util.concurrent Thread-Safe Unbounded, lock-free doubly-ended queue. Lock-free doubly-linked deque (CAS-based). Supports efficient insertion/removal from both ends concurrently. offer/poll both ends O(1) lock-free
  • Work-stealing patterns
  • Concurrent deque operations from multiple threads
  • Bounded or blocking scenarios
ArrayBlockingQueuejava.util.concurrent Thread-Safe Bounded blocking FIFO queue backed by a fixed-size array. Circular array with a single ReentrantLock + two Conditions (notEmpty, notFull). Capacity set at construction; cannot grow. Optionally fair ordering. put/take O(1) blocks on full/empty
  • Producer/consumer with bounded buffer
  • Back-pressure / flow control
  • Thread-pool work queues
  • Unbounded scenarios (use LinkedBlockingQueue)
  • Very high throughput (lock contention)
LinkedBlockingQueuejava.util.concurrent Thread-Safe Optionally bounded blocking FIFO queue backed by linked nodes. Two-lock design: separate locks for head (take) and tail (put), enabling higher concurrency than ArrayBlockingQueue. Optionally bounded (default Integer.MAX_VALUE). put/take O(1) two-lock design
  • High-throughput producer/consumer
  • Default queue for ThreadPoolExecutor
  • Unbounded use in memory-constrained apps (default is near-unlimited)
PriorityBlockingQueuejava.util.concurrent Thread-Safe Unbounded blocking min-heap queue. Binary heap with a single ReentrantLock. Unbounded (grows dynamically). Blocks only on take when empty; never blocks on put. offer O(log n) poll O(log n) peek O(1)
  • Concurrent priority scheduling
  • Task executors with priority levels
  • Bounded capacity required (unbounded)
  • When FIFO among equal-priority items is needed
LinkedBlockingDequejava.util.concurrent Thread-Safe Optionally bounded blocking deque (both-ended queue). Doubly-linked list with a single lock. All Deque operations are blocking variants. Default capacity is Integer.MAX_VALUE. put/take both ends O(1)
  • Work-stealing thread pools
  • When both head and tail blocking access is needed
  • High-throughput one-ended queues (use LinkedBlockingQueue)
SynchronousQueuejava.util.concurrent Thread-Safe Zero-capacity handoff queue: each put must wait for a take. No internal storage. Uses a stack (unfair) or queue (fair) of waiting threads. A put blocks until a corresponding take occurs — direct thread-to-thread handoff. put/take O(1) zero buffer
  • Direct thread handoff / rendezvous
  • Used internally by Executors.newCachedThreadPool()
  • Any use case requiring buffering
  • When producer rate can exceed consumer rate
DelayQueuejava.util.concurrent Thread-Safe Unbounded blocking queue where elements are released only after their delay has expired. Backed by a PriorityQueue of Delayed elements with a ReentrantLock. take() blocks until the element with the smallest remaining delay expires. offer O(log n) poll O(log n)
  • Task scheduling with future expiry
  • Cache eviction, TTL-based queues
  • Session/token expiry
  • When immediate or FIFO ordering is needed
  • Elements are not Delayed
LinkedTransferQueuejava.util.concurrent Thread-Safe Unbounded, lock-free queue combining SynchronousQueue and LinkedBlockingQueue semantics. Dual-data structure: nodes are either data nodes or reservation (waiter) nodes. CAS-based, lock-free. transfer() hands off directly to a waiting consumer; put() enqueues if no consumer waits. transfer/put O(1) lock-free
  • High-performance message passing
  • When both buffering and direct handoff are needed
  • Simple producer/consumer where LinkedBlockingQueue suffices
  • Memory-sensitive apps (unbounded)

Locks & Synchronizers

Java provides multiple synchronization primitives in java.util.concurrent.locks and java.util.concurrent. Choosing the right one prevents race conditions while maximising throughput.

⚠️ The Problem — Race Conditions

Without synchronization: two threads read-modify-write the same variable

1 · synchronized vs ReentrantLock

🔑 synchronized (built-in)

Java keyword; every object has an intrinsic monitor. JVM manages acquisition and release automatically.

synchronized (this) { // critical section counter++; } // auto-released on exit/exception
✓ Auto-release ✓ Simple syntax ✗ No timeout ✗ Not interruptible ✗ No fair mode

✅ Use when

  • Simple critical sections
  • Low contention code
  • Condition via wait/notify

🚫 Avoid when

  • Need timed or interruptible locking
  • Need fairness guarantees

🔐 ReentrantLock

j.u.c.locks

Explicit lock with the same mutual-exclusion semantics but far richer API. Re-entrant: same thread can acquire multiple times.

ReentrantLock lock = new ReentrantLock(true); // fair if (lock.tryLock(500, TimeUnit.MILLISECONDS)) { try { counter++; } finally { lock.unlock(); } }
✓ tryLock(timeout) ✓ Interruptible ✓ Fair mode ✓ Multiple Conditions ✗ Must call unlock()

✅ Use when

  • Need timed / interruptible locking
  • Need fairness or multiple wait-sets

🚫 Avoid when

  • Simple synchronized block suffices (extra overhead)

ReentrantReadWriteLock rwl = new ReentrantReadWriteLock(); Lock r = rwl.readLock(), w = rwl.writeLock(); // Reader (many at a time) r.lock(); try { return data; } finally { r.unlock(); } // Writer (exclusive) w.lock(); try { data = newVal; } finally { w.unlock(); }

✅ Use when

  • Read-heavy with occasional writes
  • Caches, config stores

🚫 Avoid when

  • Write-heavy (no benefit, extra overhead)
  • Write-lock upgrade needed (not supported)

📡 Phaser

✅ Use when

  • Complex multi-phase pipelines

🚫 Avoid when

  • Fixed parties and simple sync (CyclicBarrier is simpler)

🔢 newFixedThreadPool(n)

Creates exactly n threads that live forever. Unbounded LinkedBlockingQueue. Extra tasks queue until a thread is free.

ExecutorService ex = Executors.newFixedThreadPool(4);

✅ Use when

  • CPU-bound tasks (n = CPU cores)
  • Predictable concurrency needed

🚫 Avoid when

  • Task submission rate >> completion rate (unbounded queue → OOM)

📈 newCachedThreadPool()

Creates new threads on demand; reuses idle threads (60 s TTL). Uses SynchronousQueue — zero buffer, direct handoff.

Queue: 0
→
0 → ∞ workers
ExecutorService ex = Executors.newCachedThreadPool();

✅ Use when

  • Short-lived async tasks
  • I/O-bound tasks, high burst tolerance

🚫 Avoid when

  • Sustained high load (unbounded threads → OOM)
  • CPU-bound tasks

1️⃣ newSingleThreadExecutor()

Exactly one thread executing tasks serially. Unbounded queue. Guarantees task ordering (FIFO). Thread is replaced if it dies.

ExecutorService ex = Executors.newSingleThreadExecutor();

✅ Use when

  • Sequential background work
  • Single-writer pattern
  • Event loop simulation

🚫 Avoid when

  • Parallel throughput is required

⏰ newScheduledThreadPool(n)

Supports delayed and periodic task execution. Uses a DelayedWorkQueue (min-heap). Also: newSingleThreadScheduledExecutor().

ScheduledExecutorService s = Executors.newScheduledThreadPool(2); s.schedule(task, 5, TimeUnit.SECONDS); s.scheduleAtFixedRate(task, 0, 1, MINUTES); s.scheduleWithFixedDelay(task, 0, 500, MS);

✅ Use when

  • Periodic health-checks, polling
  • Delayed retries, cron-like jobs

🚫 Avoid when

  • Full cron semantics needed (use Quartz)

🥷 newWorkStealingPool()

Backed by a ForkJoinPool with parallelism = CPU cores. Idle threads steal tasks from busy threads' deques. Java 8+.

ExecutorService ex = Executors.newWorkStealingPool(); // or explicitly: new ForkJoinPool(Runtime.getRuntime() .availableProcessors());

✅ Use when

  • Many small independent tasks
  • Recursive divide-and-conquer

🚫 Avoid when

  • Tasks have dependencies / ordering requirements

🪶 newVirtualThreadPerTaskExecutor() Java 21

Creates a virtual thread (Project Loom) for every submitted task. Virtual threads are JVM-managed, extremely lightweight (~few KB heap), and can number in the millions.

ExecutorService ex = Executors.newVirtualThreadPerTaskExecutor(); // or per-task: Thread.ofVirtual().start(runnable);
✓ Millions of threads ✓ Blocks cheaply (no carrier pinning for I/O) ⚠ Avoid synchronized + blocking I/O (pins carrier)

✅ Use when

  • High-concurrency I/O (HTTP, DB, file)
  • Replacing thread-per-request model

🚫 Avoid when

  • CPU-bound tasks (no benefit over platform threads)
  • Native synchronized blocks inside virtual thread (pin risk)

CompletableFuture

Introduced in Java 8, CompletableFuture<T> solves the problems of blocking Future.get() and callback hell by providing a composable, non-blocking async pipeline.

The Problem — Why Future<T> Alone Is Not Enough

❌ Old Way: Future<T>

Future<User> fu = executor.submit(() -> fetchUser()); Future<Order> fo = executor.submit(() -> fetchOrders()); User u = fu.get(); // 🛑 BLOCKS thread! Order o = fo.get(); // 🛑 BLOCKS again enrich(u, o); // can't pipeline
✗ get() blocks the calling thread ✗ No chaining / composition ✗ No exception handling pipeline ✗ No combine multiple futures

✅ New Way: CompletableFuture<T>

CompletableFuture.supplyAsync(() -> fetchUser()) .thenCombine( CompletableFuture.supplyAsync(() -> fetchOrders()), (user, orders) -> enrich(user, orders) ) .thenAccept(result -> send(result)) .exceptionally(ex -> { logError(ex); return null; });
✓ Non-blocking pipeline ✓ Composable stages ✓ Built-in error handling ✓ Combine multiple futures

Method Taxonomy — What Each Stage Does

🚀 Create / Supply

// Start async computation CF.supplyAsync(supplier) CF.supplyAsync(supplier, executor) CF.runAsync(runnable) // Wrap a known value CF.completedFuture(value) CF.failedFuture(ex)

🔀 Transform (1-to-1)

// thenApply = map (sync in same thread) cf.thenApply(t -> transform(t)) cf.thenApplyAsync(t -> transform(t)) // thenCompose = flatMap (returns CF<U>) cf.thenCompose(t -> asyncCall(t)) // thenRun = fire-and-forget cf.thenRun(() -> log("done"))

🔗 Combine (2 futures)

// Both must complete → combine values cf1.thenCombine(cf2, (a,b) -> merge(a,b)) // Both complete → side effect only cf1.thenAcceptBoth(cf2, (a,b) -> save(a,b)) // Wait for EITHER — first wins cf1.applyToEither(cf2, t -> transform(t)) cf1.acceptEither(cf2, t -> use(t))

📋 Collect (N futures)

// Wait for ALL to complete CF.allOf(cf1, cf2, cf3) .thenRun(() -> process()) // Complete as soon as ONE finishes CF.anyOf(cf1, cf2, cf3) .thenAccept(first -> use(first))

🛡️ Error Handling

// Recover from exception → provide fallback cf.exceptionally(ex -> fallback) // See result OR exception (like finally) cf.handle((t, ex) -> ex != null ? fallback : t) // Side-effect always (doesn't change value) cf.whenComplete((t, ex) -> log(t, ex))

⚙️ Async Variants

Every thenXxx has an thenXxxAsync variant that runs the callback in a different thread:

// No suffix → same thread as predecessor cf.thenApply(fn) // Async → ForkJoinPool.commonPool() cf.thenApplyAsync(fn) // Async → custom executor cf.thenApplyAsync(fn, myExecutor)
⚠️ Avoid chaining blocking calls without Async — they block the completing thread.
💡 thenApply vs thenCompose: Use thenApply(t → U) when the transformation is synchronous. Use thenCompose(t → CF<U>) when the next step is itself async (like another supplyAsync). thenCompose is the flatMap of CompletableFuture and avoids nested CF<CF<T>>.

Fork/Join Framework

Introduced in Java 7, the Fork/Join framework is designed for divide-and-conquer parallelism: recursively split a large task into smaller sub-tasks, execute them in parallel, then merge the results.

Core Concepts

🏗️ ForkJoinPool

Specialized thread pool that uses work-stealing. Default: ForkJoinPool.commonPool() (parallelism = CPU cores − 1).

ForkJoinPool pool = ForkJoinPool.commonPool(); // or custom parallelism: ForkJoinPool pool = new ForkJoinPool(4);

📊 RecursiveTask<T>

A task that returns a value. Override compute() to fork sub-tasks and join their results.

class SumTask extends RecursiveTask<Long> { @Override protected Long compute() { if (size <= THRESHOLD) return sumDirect(); SumTask left = new SumTask(lo, mid); SumTask right = new SumTask(mid, hi); left.fork(); // async submit left return right.compute() // run right inline + left.join(); // wait for left } }

🔧 RecursiveAction

A task that returns void. Used for side-effect operations like filling, sorting, or updating arrays in place.

class FillAction extends RecursiveAction { @Override protected void compute() { if (size <= THRESHOLD) { Arrays.fill(arr, lo, hi, value); return; } FillAction left = new FillAction(lo, mid); FillAction right = new FillAction(mid, hi); invokeAll(left, right); // fork+join both } }
Decomposition Tree — Parallel Sum of array [1..8]

Task splits recursively until it hits the threshold (≤ 2 elements), then merges results bottom-up

sum(1..8) = 36
sum(1..4) = 10
sum(5..8) = 26
sum(1,2)=3
sum(3,4)=7
sum(5,6)=11
sum(7,8)=15
↑ Leaf tasks execute on separate CPU cores simultaneously → results bubble up via join()
Key pattern: always fork() the left sub-task (pushes it to the thread's deque), then compute() the right inline (reuses current thread), then join() the left. This avoids creating an unnecessary thread switch.
Full Example — Parallel Sum with RecursiveTask

RecursiveTask<Long> — returns sum

import java.util.concurrent.*; class ParallelSum extends RecursiveTask<Long> { static final int THRESHOLD = 1000; private final int[] arr; private final int lo, hi; ParallelSum(int[] arr, int lo, int hi) { this.arr=arr; this.lo=lo; this.hi=hi; } @Override protected Long compute() { if (hi - lo <= THRESHOLD) { // base case long sum = 0; for (int i=lo; i<hi; i++) sum += arr[i]; return sum; } int mid = (lo + hi) / 2; ParallelSum left = new ParallelSum(arr, lo, mid); ParallelSum right = new ParallelSum(arr, mid, hi); left.fork(); // async long rightResult = right.compute(); // sync return rightResult + left.join(); // merge } public static void main(String[] args) { int[] data = new int[10_000_000]; Arrays.fill(data, 1); ForkJoinPool pool = ForkJoinPool.commonPool(); long total = pool.invoke(new ParallelSum(data,0,data.length)); // total == 10_000_000 } }

RecursiveAction — parallel array fill (no return)

class ParallelFill extends RecursiveAction { static final int THRESHOLD = 500; private final int[] arr; private final int lo, hi, val; ParallelFill(int[] arr, int lo, int hi, int val) { this.arr=arr; this.lo=lo; this.hi=hi; this.val=val; } @Override protected void compute() { if (hi - lo <= THRESHOLD) { Arrays.fill(arr, lo, hi, val); return; } int mid = (lo + hi) / 2; invokeAll( new ParallelFill(arr, lo, mid, val), new ParallelFill(arr, mid, hi, val) ); // forks both, waits for both } } // Usage: int[] data = new int[1_000_000]; ForkJoinPool.commonPool() .invoke(new ParallelFill(data, 0, data.length, 42));
Work-Stealing — How ForkJoinPool Maximises CPU Utilisation

Each thread has its own deque; idle threads steal from the TAIL of busy threads' deques

🧵 Thread-1 (busy)

T1 ▶ running
T3 pending
T5 pending
T7 pending ← TAIL
⟵ steal

🧵 Thread-2 (idle)

(empty deque)
T7 stolen! ▶

🧵 Thread-3 (busy)

T4 pending

Head vs Tail: Each thread pushes/pops from its own deque's HEAD (LIFO — maintains locality). Thieves steal from the TAIL (FIFO — take oldest, coarser work), minimising synchronisation contention between owner and thief.
When to Use vs Avoid

✅ Good Fit for Fork/Join

  • CPU-bound recursive algorithms — merge sort, quicksort, matrix multiply
  • Large array/collection processing — parallel sum, max, filter
  • Tasks that naturally decompose into independent sub-tasks
  • When sub-task granularity can be tuned via a threshold
  • Arrays.parallelSort(), Stream.parallel() use it internally

🚫 Poor Fit for Fork/Join

  • I/O-bound tasks — threads block waiting → no CPU to steal (use virtual threads)
  • Tasks with shared mutable state between sub-tasks
  • Tasks that can't be split into independent units
  • When tasks are few and large — no benefit from splitting
  • When synchronized or blocking locks are used inside tasks (prevents work-stealing)
💡 Threshold Tuning: If the threshold is too small, the overhead of creating/joining tasks dominates. Too large, and you lose parallelism. A common starting point is array.length / (4 × availableProcessors). Always benchmark with your specific workload and JVM flags.
